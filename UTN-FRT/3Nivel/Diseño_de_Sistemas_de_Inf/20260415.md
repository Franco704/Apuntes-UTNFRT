
# Metodologías Tradicionales

## Proceso de Software:
Una serie de actividades relacionadas que se llevan a cabo para elaborar un producto de software. 
Análisis, Desarrollo,Validación y evolución o mantenimiento
Puede ser desarrollar software desde cero o extender y modificar sistema existentes.
No es un proceso sencillo, es aquel proceso que hacemos con software industrial o comercial, no el personal.
Este proceso implica configurar e integrar software comercial o componentes del sistema.
Los procesos de software son complejos, difícilmente se lleve a cabo por un amateur.

> [!NOTE] Recordar
> No hay un proceso ideal.

## Los procesos de software según sommervile se clasifican en dos:
1. **Dirigidos por un plan (plandriven**): todas las actividades del proceso se planean por anticipado y el avance se mide contra dicho plan.
2. **Procesos ágiles**: la planeación es **incremental**. Es más fácil modificar el proceso para reflejar los requerimientos cambiantes del cliente. Corregir un incremento no es una funcionalidad más, no es un incremento

Cada enfoque es adecuado para diferentes tipos de software, hay que buscar un equilibrio entre procesos dirigidos por un plan y procesos ágiles.
No hay un proceso de software "ideal", siempre podemos mejorar el proceso de software.


## Modelos de proceso de software:

Representación simplificada del proceso de desarrollo de software.
Sommerville enseña 3 Modelos del Proceso:
1. El modelo en cascada (waterfall)
2. Desarrollo Incremental
3. Ingenieria de software orientada a la reutilización.

Estos modelos se utilizan en conjunto para desarrollar grandes sistemas.
Nos permite obtener las mejores caracteristicas de los modelos.
Para diseñar la arquitectura del sistema necesitamos información sobre sus requerimientos esenciales, por lo tanto no se pueden hacer en forma incremental.
Los subsistemas dentro de un sistema más grande se pueden desarrollar con diferentes enfoques.
Partes del sistema que son bien comprendidas pueden especificarse y desarrollarse mediante un proceso basado en cascada.
Partes del sistema dificiles de especificar, como la interfaz de usuario, deben desarrollarse con enfoque incremental.

## El modelo en cascada o de Ciclo de Vida del Sistema
Toma las actividades fundamentales del proceso y las representa como fases separadas.
- Definición de requerimientos
- Diseño del sistema y del software
- Implementación y prueba de unidad
- Integración y prueba del sistema
- Operación y mantenimiento

Fue el primer modelo del proceso de desarrollo de software
Por el paso de una fase en cascada a otra, se conoce como "modelo en cascada"
Es un proceso dirigido por un plan, se planean y programan todas las actividades del proceso antes de comenzar a trabajar.
Sus etapas reflejan las actividades fundamentales del desarrollo
El resultado de cada fase consiste en uno o más documentos que se autorizaron o firmaron.
La siguiente fase no debe comenzar sino hasta que termine la fase previa.
En la práctica las etapas se solapan y se nutren mutuamente de información.
Durante el diseño se identifican problemas del analisis, en el codigo se descubren problemas de diseño y asi.

> [!NOTE]RECORDAR
> El proceso de software no es un modelo lineal, hay retroalimentacion de una fase a otra

Las iteraciones son caras e implican un rediseño significativo,
El sistema debe evolucionar para mantenerse útil.
Hacer tales cambios (mantenimiento) significa rehacer etapas anteriores.


## Desarrollo Incremental
Se basa en diseñar una implementacion inicial, exponerla al comentario del usuario, y luego desarrollarla, en sus diversas versiones, hasta producir el sistema.
Las actividades de especificación, desarrollo y validación están entrelazadas, en vez de separadas, con rápida retroalimentación.

No hay una division tagante en las tareas como en el anterior modelo.
No se puede planificar mucho, no se tiene conciencia exacta de los tiempos
El desarrollo incremental, es mejor que un enfoque en cascada para sistemas empresariales, de comercio electrónico y personales.
El desarrollo incremental refleja la forma en que se resuelven problemas.
No se hace por adelantado una solución y se retrocede cuando se detectan errores.
Así es más barato y fácil realizar cambios en el software conforme se diseña

> [!NOTE]RECORDAR
> Cada incremento o versión del sistema incorpora alguna función que necesita el cliente.

Los primeros incrementos incluyen la función más importante o la más urgente, así el cliente puede evaluar el desarrollo del sistema en una etapa temprana, para constatar si se entrega lo que requiere.
En caso contrario, debe cambiarse el incremento actual y definir una nueva función para incrementos posteriores.
Es el enfoque más común para el desarrollo de sistemas.
Este enfoque puede estar basado en un plan, ser ágil o, una mezcla de ambos.
En un enfoque basado e un plan se identifican por adelantado los incrementos.
Si se adopta un enfoque ágil, se detectan los primeros incrementos, y el desarrollo de incrementos posteriores depende del avance y prioridades del cliente.

Desarrollo Incremental Vs Modelo en Cascada
1. En el DI es menor el costo de adaptar los requerimientos cambiante del cliente. El análisis y la documentación que tiene que reelaborarse es menor que lo requerido en el MC.
2. En el DI es más sencillo obtener retroalimentacion de cliente, que puede comentar las demostraciones del software y darse cuenta de cuanto se ha implementado. Es dificil para el cliente juzgar el avance solo con documentos de diseño.
3. En el DI es más rapida la entrega e implementación de software til al cliente, aun sin toda la funcionalidad. LOs clientes pueden usar el software más temprano.

Problemas del enfoque incremental
1. El proceso no es visible. Los administradores necesitan entregas regulares para medir el avance. Si los sistemas se desarrollan rápido, no vale la pena producir documentos que reflejen cada versión del sistema.
2. La estructura del sistema tiende a degradarse con más incrementos. La incorporación de más cambios se vuelve cada vez más difícil y costosa. Los problemas son particularmente agudos para sistemas grandes, complejos y de larga duración, donde diversos equipos desarrollan diferentes partes del sistema.

Ingenieria del Software orientada a la reutilización

EN muchos proyectos hay reutilizacion de software, puede ser de manera informal (los desarrolladores conocen diseños o codigos similares) Los buscan, modifican segun se necesite e incoporan en sus sistemas.
Hoy la reutilizacion se usa ampliamente

La etapa inicial de Especificacion de Requerimientos y la de Validación son como otros procesos.
Pero las etapas intermedias son diferentes

Etapas Intermedias:
1. Analisis de COmponentes: Especificados los requerimientos (Análisis), buscamos componentes para implementar esa especificacion, Por lo general, no hay coincidencia exacta y los componentes que se usan proporcionan solo parte de la funcionalidad requerida.
2. Modificacion de Requerimientos: Analizamos los requerimientos con la informacion de los componentes descubiertos y los modificamos para reflejar lo que necesitamos. Si las modificaciones son imposibles, volvemos a la etapa anterior (Analisis de Componentes) para buscar soluciones alternativas.
3. Diseño de Sistema con Reutilizacion: Diseñamos le maro conceptual del sistema con los componentes reutilizados.
4. Desarrollo e integración: Diseñamos el software que no puede procurarse de manera externa, y se integran los componentes y los sistemas adquiridos para crear el nuevo sistema. Implica también la integración de todo el sistema, en este modelo.


Ventajas de la Ing de Soft orientada a la reutilizacion
- Reduce la cantida de software a desarrollar
- DIsminuye costos y riesgos
- Se consiguen entregas más rapidas
Desventajas
- El sistema reutilizado puede no cubrir las neesidades reales de los usuarios.
- Se pierde control sobre la evolución del sistema.

Integración de Sistemas
El desarrollo de sistemas incluye: especificación de requerimientos, diseño, construcción, integración y pruebas. Los sistemas grandes y complejos consisten por lo general de una mezcla de componentes comerciales y especialmente construidos.
EN grandes desarrollos el proveedor del sistema, suele subcontratar el desarrollo de diferentes subsistemas con subcontratistas.
Integración del sistema: los componentes se reúnen para crear un nuevo sistema. Solo entonces se vuelven evidentes las propiedades emergentes del sistema.
La integracion del sistema, donde los subsistemas con más de un proveedor deben reunirse para trabajar juntos, es particularmente critica.
Durante la integración de sistemas se toman los subsistemas desarrollados de manera independiente y se reúnen para constituir un sistema completo.
Esta integración se puede realizar bajo un enfoque de "big-bang" todos los subsistemas se integran al mismo tiempo, aunque por razones técnicas y administrativas, la mejor opción es un proceso de integración incremental a que los subsistemas se integren uno a la vez.

Razónes:
1. Es casi imposible programar que todos los subsistemas se terminen al mismo tiempo.
2. La integración incremental reduce el costo de localizar errores.

La etapa final del proceso de desarorollo de sistema es la entrega y la implementacion del sistema
