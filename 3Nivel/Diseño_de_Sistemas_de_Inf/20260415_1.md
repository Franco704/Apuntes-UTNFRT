# Metodologías Tradicionales

## Proceso de Software
Una serie de actividades relacionadas que se llevan a cabo para elaborar un producto de software: Análisis, Desarrollo, Validación y evolución o mantenimiento.
Puede ser desarrollar software desde cero o extender y modificar sistemas existentes.
No es un proceso sencillo, es aquel proceso que hacemos con software industrial o comercial, no el personal. Este proceso implica configurar e integrar software comercial o componentes del sistema.
Los procesos de software son complejos, difícilmente se lleve a cabo por un amateur.

> [!NOTE] Recordar
> No hay un proceso ideal.

### Clasificación según Sommerville:
1. **Dirigidos por un plan (plan-driven):** todas las actividades del proceso se planean por anticipado y el avance se mide contra dicho plan.
2. **Procesos ágiles:** la planeación es **incremental**. Es más fácil modificar el proceso para reflejar los requerimientos cambiantes del cliente. Corregir un incremento no es una funcionalidad más, no es un incremento.

Cada enfoque es adecuado para diferentes tipos de software, hay que buscar un equilibrio entre procesos dirigidos por un plan y procesos ágiles. No hay un proceso de software "ideal", siempre podemos mejorar el proceso de software.

## Modelos de proceso de software

Representación simplificada del proceso de desarrollo de software.
Sommerville enseña 3 Modelos del Proceso:
1. El modelo en cascada (waterfall)
2. Desarrollo Incremental
3. Ingeniería de software orientada a la reutilización.

Estos modelos se utilizan en conjunto para desarrollar grandes sistemas. Nos permite obtener las mejores características de los modelos.
Para diseñar la arquitectura del sistema necesitamos información sobre sus requerimientos esenciales, por lo tanto no se pueden hacer en forma incremental.
Los subsistemas dentro de un sistema más grande se pueden desarrollar con diferentes enfoques. Partes del sistema que son bien comprendidas pueden especificarse y desarrollarse mediante un proceso basado en cascada. Partes del sistema difíciles de especificar, como la interfaz de usuario, deben desarrollarse con enfoque incremental.

### 1. El modelo en cascada o de Ciclo de Vida del Sistema
Toma las actividades fundamentales del proceso y las representa como fases separadas:
* Definición de requerimientos
* Diseño del sistema y del software
* Implementación y prueba de unidad
* Integración y prueba del sistema
* Operación y mantenimiento

Fue el primer modelo del proceso de desarrollo de software. Por el paso de una fase en cascada a otra, se conoce como "modelo en cascada". Es un proceso dirigido por un plan, se planean y programan todas las actividades del proceso antes de comenzar a trabajar.
Sus etapas reflejan las actividades fundamentales del desarrollo. El resultado de cada fase consiste en uno o más documentos que se autorizaron o firmaron. La siguiente fase no debe comenzar sino hasta que termine la fase previa.
En la práctica las etapas se solapan y se nutren mutuamente de información. Durante el diseño se identifican problemas del análisis, en el código se descubren problemas de diseño y así.

> [!WARNING] Recordar
> El proceso de software no es un modelo lineal, hay retroalimentación de una fase a otra.

Las iteraciones son caras e implican un rediseño significativo. El sistema debe evolucionar para mantenerse útil. Hacer tales cambios (mantenimiento) significa rehacer etapas anteriores.

### 2. Desarrollo Incremental
Se basa en diseñar una implementación inicial, exponerla al comentario del usuario, y luego desarrollarla, en sus diversas versiones, hasta producir el sistema. Las actividades de especificación, desarrollo y validación están entrelazadas, en vez de separadas, con rápida retroalimentación.

No hay una división tajante en las tareas como en el anterior modelo. No se puede planificar mucho, no se tiene conciencia exacta de los tiempos.
El desarrollo incremental, es mejor que un enfoque en cascada para sistemas empresariales, de comercio electrónico y personales. Refleja la forma en que se resuelven problemas.
No se hace por adelantado una solución y se retrocede cuando se detectan errores. Así es más barato y fácil realizar cambios en el software conforme se diseña.

> [!TIP] Recordar
> Cada incremento o versión del sistema incorpora alguna función que necesita el cliente.

Los primeros incrementos incluyen la función más importante o la más urgente, así el cliente puede evaluar el desarrollo del sistema en una etapa temprana, para constatar si se entrega lo que requiere. En caso contrario, debe cambiarse el incremento actual y definir una nueva función para incrementos posteriores.
Es el enfoque más común para el desarrollo de sistemas. Puede estar basado en un plan, ser ágil o, una mezcla de ambos. En un enfoque basado en un plan se identifican por adelantado los incrementos. Si se adopta un enfoque ágil, se detectan los primeros incrementos, y el desarrollo de incrementos posteriores depende del avance y prioridades del cliente.

**Desarrollo Incremental Vs Modelo en Cascada**
1. En el DI es menor el costo de adaptar los requerimientos cambiante del cliente. El análisis y la documentación que tiene que reelaborarse es menor que lo requerido en el MC.
2. En el DI es más sencillo obtener retroalimentación del cliente, que puede comentar las demostraciones del software y darse cuenta de cuánto se ha implementado. Es difícil para el cliente juzgar el avance solo con documentos de diseño.
3. En el DI es más rápida la entrega e implementación de software útil al cliente, aun sin toda la funcionalidad. Los clientes pueden usar el software más temprano.

**Problemas del enfoque incremental**
1. El proceso no es visible. Los administradores necesitan entregas regulares para medir el avance. Si los sistemas se desarrollan rápido, no vale la pena producir documentos que reflejen cada versión del sistema.
2. La estructura del sistema tiende a degradarse con más incrementos. La incorporación de más cambios se vuelve cada vez más difícil y costosa. Los problemas son particularmente agudos para sistemas grandes, complejos y de larga duración, donde diversos equipos desarrollan diferentes partes del sistema.

### 3. Ingeniería del Software orientada a la reutilización
En muchos proyectos hay reutilización de software, puede ser de manera informal (los desarrolladores conocen diseños o códigos similares). Los buscan, modifican según se necesite e incorporan en sus sistemas. Hoy la reutilización se usa ampliamente.

La etapa inicial de Especificación de Requerimientos y la de Validación son como otros procesos. Pero las etapas intermedias son diferentes.

**Etapas Intermedias:**
1. **Análisis de Componentes:** Especificados los requerimientos (Análisis), buscamos componentes para implementar esa especificación. Por lo general, no hay coincidencia exacta y los componentes que se usan proporcionan solo parte de la funcionalidad requerida.
2. **Modificación de Requerimientos:** Analizamos los requerimientos con la información de los componentes descubiertos y los modificamos para reflejar lo que necesitamos. Si las modificaciones son imposibles, volvemos a la etapa anterior (Análisis de Componentes) para buscar soluciones alternativas.
3. **Diseño de Sistema con Reutilización:** Diseñamos el marco conceptual del sistema con los componentes reutilizados.
4. **Desarrollo e integración:** Diseñamos el software que no puede procurarse de manera externa, y se integran los componentes y los sistemas adquiridos para crear el nuevo sistema. Implica también la integración de todo el sistema, en este modelo.

**Ventajas:**
* Reduce la cantidad de software a desarrollar.
* Disminuye costos y riesgos.
* Se consiguen entregas más rápidas.

**Desventajas:**
* El sistema reutilizado puede no cubrir las necesidades reales de los usuarios.
* Se pierde control sobre la evolución del sistema.

## Integración de Sistemas
El desarrollo de sistemas incluye: especificación de requerimientos, diseño, construcción, integración y pruebas. Los sistemas grandes y complejos consisten por lo general de una mezcla de componentes comerciales y especialmente construidos.
En grandes desarrollos el proveedor del sistema suele subcontratar el desarrollo de diferentes subsistemas con subcontratistas.

**Integración del sistema:** los componentes se reúnen para crear un nuevo sistema. Solo entonces se vuelven evidentes las propiedades emergentes del sistema.
La integración del sistema, donde los subsistemas con más de un proveedor deben reunirse para trabajar juntos, es particularmente crítica.
Durante la integración de sistemas se toman los subsistemas desarrollados de manera independiente y se reúnen para constituir un sistema completo.

Esta integración se puede realizar bajo un enfoque de "big-bang" (todos los subsistemas se integran al mismo tiempo), aunque por razones técnicas y administrativas, la mejor opción es un proceso de integración incremental a que los subsistemas se integren uno a la vez.

**Razones para la integración incremental:**
1. Es casi imposible programar que todos los subsistemas se terminen al mismo tiempo.
2. La integración incremental reduce el costo de localizar errores.

La etapa final del proceso de desarrollo de sistema es la entrega y la implementación del sistema.
